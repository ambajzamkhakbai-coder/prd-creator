## 概述

本指南为 Step 1「讨论需求逻辑」提供详细的操作指导。需求逻辑讨论是整个PRD创建流程的基础，目标是明确MVP要做什么、为什么做、怎么做。

---

## Step 1.1 需求范围确定的操作指南

### 信息收集要点

从用户的描述中提取以下关键信息：

| 信息类型 | 说明 | 示例 |
|---|---|---|
| 产品背景 | 为什么要做这个产品/功能 | "现有系统无法满足社区团购需求" |
| 目标用户 | 产品服务于谁 | "社区团长和社区居民" |
| MVP设定逻辑 | 为什么选择这些功能作为MVP | "先验证团购模式可行性" |
| 设计思路 | 整体的产品方向和理念 | "轻量级、社交驱动" |
| 已知约束 | 时间、资源、技术限制 | "3周内上线、仅微信小程序" |

### 需求范围输出规范

输出MVP需求范围时，必须包含以下内容：

1. **纳入MVP的功能清单**：每个功能点附带纳入理由
2. **暂不纳入的功能清单**：每个功能点附带暂缓原因
3. **功能优先级排序**：P0（必须有）/ P1（最好有）/ P2（可以后加）
4. **范围边界说明**：明确什么在范围内、什么不在

### 需求范围判断原则

- MVP聚焦核心价值验证，不追求功能完备
- 优先选择能形成完整用户闭环的功能
- 依赖外部系统或大量开发的功能可考虑暂缓
- 用户明确表示重要的功能必须纳入

### 优先级确认环节

输出需求范围与优先级总览后，**必须显式向用户发起确认**：

1. 展示「优先级判断理由」列与「优先级排序总览」
2. 提问："以上是基于启发式给出的优先级建议，是否需要调整某个功能的优先级？"
3. 收到调整请求后，按用户给出的「功能编号 → 目标优先级」更新表格与排序总览
4. 全部确认后再进入 Step 1.2

## P0/P1/P2 启发式判断规则

为每个纳入 MVP 的功能自动标注优先级，并在「优先级判断理由」列写明依据。

### 分级定义

| 优先级 | 定义 | 判断关键问题 |
|---|---|---|
| **P0** | 核心闭环必需。缺它产品无法完成主流程或无法验证核心假设 | "拿掉这个功能，用户还能完成核心目标吗？" → 不能 = P0 |
| **P1** | 显著体验提升但非必需。缺它产品能跑通，但留存/转化/效率会明显打折 | "拿掉它产品还能跑，但关键指标会明显下降吗？" → 是 = P1 |
| **P2** | 锦上添花。缺它对核心指标几乎无影响，属于扩展或差异化点 | "拿掉它对核心闭环和关键指标几乎没影响？" → 是 = P2 |

### 判断启发式

按以下问题逐项排查，命中靠前问题的优先级更高：

1. **是否处于核心闭环关键节点？**（如电商的"下单/支付"、内容产品的"创作/分发"）→ P0
2. **缺它会导致核心数据无法采集或核心假设无法验证？** → P0
3. **缺它会让首日留存/转化率明显下滑？**（如登录注册、消息通知）→ P1
4. **缺它会让某类用户体验明显受损但不影响主流程？**（如搜索、筛选、收藏）→ P1
5. **属于差异化点、社交分享、个性化配置？** → P2
6. **属于"等数据起来再做"的运营/管理后台扩展？** → P2

### 典型示例

| 产品类型 | P0 示例 | P1 示例 | P2 示例 |
|---|---|---|---|
| 电商 MVP | 商品浏览、下单、支付、订单查询 | 登录注册、收货地址管理、订单状态推送 | 收藏、分享、优惠券 |
| 内容社区 MVP | 注册登录、发布、Feed流、内容详情 | 评论、点赞、消息提醒 | 关注关系、话题、勋章 |
| 工具类 MVP | 主功能输入与产出、结果保存 | 历史记录、账号同步 | 主题切换、社交分享 |

### 输出规范

每条功能的「优先级判断理由」列要给出**简短但具体**的依据，避免泛泛而谈：

- ❌ 不好："核心功能，必须有"
- ✅ 好："P0，没有商品浏览用户进来无法触达任何商品，整个购买闭环断裂"
- ✅ 好："P1，没有消息推送用户可以手动刷新查看，但流失率预计上升 20%+"

## Step 1.2 需求逻辑定义的操作指南

### 模块拆分原则

将MVP需求范围拆分为逻辑模块时，遵循以下原则：

1. **按用户目标拆分**：每个模块对应一个用户目标或任务
2. **高内聚低耦合**：模块内部功能紧密相关，模块间依赖尽量少
3. **粒度适中**：一个模块包含3-8个功能点为宜
4. **命名清晰**：模块名称能直观反映其职责

### 逐模块讨论流程

对每个模块执行以下步骤：

```
1. 告知用户当前模块名称和目标
2. 应用思考框架（见下方）
3. 向用户提出针对性问题
4. 收集用户回答
5. 组织模块需求逻辑内容
6. 检查是否有模糊点，有则追问
7. 展示整理后的模块内容
8. 用户确认 → 进入下一模块
```

### 模块间依赖处理

- 讨论新模块时，回顾已确认模块中与之相关的逻辑
- 如果发现模块间存在依赖关系，明确标注
- 如果发现与已确认模块的逻辑冲突，立即指出并与用户讨论解决

## 模块讨论的思考框架

对每个模块，按以下框架思考并向用户提问：

### 第一层：讲清楚的标准

思考："怎么才算讲清楚了 [模块名] 这件事情？"

判断标准：
- 用户是谁、在什么场景下使用
- 用户要完成什么目标
- 有哪些操作步骤
- 每步操作的输入和输出是什么
- 异常情况如何处理

### 第二层：需要的信息

思考："如果要讲清楚这件事情，我需要告诉用户什么信息？"

按维度收集信息：

| 维度 | 关键问题 |
|---|---|
| 用户与场景 | 谁用？什么时候用？为什么用？ |
| 核心流程 | 主要操作步骤是什么？顺序是什么？ |
| 数据与状态 | 涉及哪些数据？状态如何流转？ |
| 规则与约束 | 有什么业务规则？什么条件下不允许操作？ |
| 异常与边界 | 出错了怎么办？极端情况怎么处理？ |

### 第三层：组织内容

收集到足够信息后，按以下结构组织模块内容：

```markdown
### [模块名]

#### 模块目标
[一句话说明这个模块要解决什么问题]

#### 核心功能点
1. [功能点1]：[说明]
2. [功能点2]：[说明]

#### 交互逻辑
[用户操作流程描述]

#### 后端逻辑
[数据处理、状态流转、业务规则]

#### 边界条件与异常处理
[异常场景及处理方式]
```

## 列表与筛选表单默认排序规则指南

### 为什么要在需求阶段就给出默认排序规则

排序规则是产品体验的"隐性契约"。很多 PRD 只写"展示订单列表"就了事，交付时才发现前端按 id 顺序、后端按插入顺序、测试按更新时间——三方各说各话。默认排序规则如果留到开发阶段再补，会带来三个必然后果：接口重构、索引反复加删、用户投诉"为什么我刚提交的排在最后"。

因此，只要模块内出现列表、表格、Feed、排行榜等含"多条数据同屏展示"的场景，就必须在需求逻辑定义阶段一次性讲清楚默认排序规则，让开发无需二次推敲即可落地。

### 触发场景识别清单

模块内**出现以下任一场景**时，必须显式给出默认排序规则：

| 场景类型 | 典型示例 |
|---|---|
| 明细列表 / 分页列表 | 订单列表、消息列表、商品列表、员工列表 |
| 后台管理表格 | 用户管理、审核工单、日志查询 |
| 搜索结果页 | 商品搜索、内容搜索、用户搜索 |
| 带筛选表单的结果集 | 高级筛选后的候选列表 |
| Feed 流 / 时间线 | 朋友圈、动态、通知中心 |
| 排行榜 / TOP N | 销量榜、积分榜、热搜榜 |
| 下拉选项 / 标签集合 | 城市选择器、分类树、标签墙 |
| 关联对象列表 | 评论、回复、附件、参与人 |

### 默认排序规则的五要素

每条默认排序规则**必须完整包含以下五点**，缺一即视为规则不完整：

| 要素 | 说明 | 反例 | 正例 |
|---|---|---|---|
| **主排序字段** | 用哪个字段作为一级排序键 | "按最新排序" | `created_at`（订单创建时间） |
| **排序方向** | `DESC` 或 `ASC`，并说明理由 | 只写"排序" | `DESC`，最新在前，符合用户"关注新事件"的心智 |
| **平局兜底字段** | 主字段相同时的二级排序键，通常用主键保证稳定 | 未定义 | 二级排序 `id DESC`，避免同一秒创建的订单前后跳动 |
| **空值处理** | `NULL` 值排在头/尾 / 视作某默认值 | 未提及 | `NULLS LAST`（例如未评分商品排在评分商品之后） |
| **可切换维度** | 用户能否切换排序，可选项列表 | 未提及 | 用户可切换"综合/销量/价格升/价格降/最新"，默认"综合" |

### 排序发生位置的判断

| 场景 | 位置 | 理由 |
|---|---|---|
| 分页列表（服务端分页） | **服务端** | 客户端无法看到全量数据，必须服务端排序后再分页 |
| 一次性加载全部（如 <200 条的下拉、标签） | 客户端 | 减少接口负担，便于前端交互切换 |
| 需要跨字段复合权重（如"综合排序"） | 服务端 | 涉及打分公式，前端难以维护一致性 |
| 用户切换排序维度 | 服务端 | 每次切换重新请求，配合分页更清晰 |

**规则**：只要涉及分页、跨字段权重、涉及权限过滤，一律**服务端排序**；否则再评估客户端排序的可行性。

### 按业务类型的默认排序策略库

以下策略可**直接套用**，无需从零推敲。选定策略后按五要素填写即可。

#### 1. 时间序列型（消息、通知、日志、动态）

- **默认字段**：`created_at`（或 `event_time`）
- **方向**：`DESC`
- **平局兜底**：`id DESC`
- **空值处理**：无空值（时间字段应 NOT NULL）
- **可切换维度**：一般不开放切换
- **场景**：站内信、系统通知、操作日志、用户动态、聊天记录（注意：聊天窗口内展示时通常再倒序为 ASC 便于阅读）
- **索引建议**：`(created_at DESC, id DESC)` 或利用主键自增性

#### 2. 未读优先型（消息中心、待办、审批工单）

- **默认字段**：先按 `is_read ASC`（未读=0排前），再按 `created_at DESC`
- **平局兜底**：`id DESC`
- **空值处理**：`is_read` 无空值
- **可切换维度**：可切换"全部/仅未读"筛选，但排序不切换
- **场景**：消息中心、待处理工单、审批列表
- **注意**：如果强调紧急程度，可再加 `priority DESC` 作为一级

#### 3. 订单/交易型（订单列表、退款单、账单）

- **默认字段**：`created_at DESC`（下单时间倒序）
- **平局兜底**：`order_id DESC`
- **空值处理**：无空值
- **可切换维度**：状态筛选（待付款/待发货/已完成等）+ 时间范围筛选。排序维度通常固定
- **场景**：C 端订单列表、B 端订单管理
- **额外规则**：若产品有"关注"或"催单"动作，被催的订单可临时置顶（`is_urgent DESC` 作为一级）

#### 4. 电商商品/搜索结果型

- **默认字段**：复合权重 `sort_score`（销量、评分、上新、库存、竞价等加权）
- **方向**：`DESC`
- **平局兜底**：`sales_count DESC`, `id DESC`
- **空值处理**：`sort_score IS NULL` 的商品置底
- **可切换维度**（必开放）：
  - 综合（默认，走 `sort_score`）
  - 销量（`sales_count DESC`）
  - 价格升序（`price ASC, id DESC`）
  - 价格降序（`price DESC, id DESC`）
  - 最新（`created_at DESC, id DESC`）
  - 评分（`rating DESC, rating_count DESC, id DESC`）
- **场景**：商品列表页、搜索结果页、类目页

#### 5. 内容社区 Feed 型

- **默认字段**：热度分数 `hot_score`（点赞、评论、转发、时间衰减复合）
- **方向**：`DESC`
- **平局兜底**：`created_at DESC, id DESC`
- **空值处理**：无
- **可切换维度**：
  - 推荐（默认，`hot_score DESC`）
  - 最新（`created_at DESC`）
  - 关注（仅关注用户 + `created_at DESC`）
- **场景**：微博式 Feed、社区首页

#### 6. 评论/回复型

- **默认字段**：`hot_score DESC`（点赞 + 回复数）或 `created_at DESC`
- **平局兜底**：`id DESC`
- **可切换维度**：
  - 最热（默认，`hot_score DESC`）
  - 最新（`created_at DESC`）
  - 楼主置顶（若有作者标记：`is_pinned DESC` 作为一级）
- **场景**：文章评论、视频评论

#### 7. 后台管理表格型

- **默认字段**：`updated_at DESC`（最近有变更的排在前面，便于运营跟进）
- **平局兜底**：`id DESC`
- **空值处理**：`updated_at NULLS LAST`
- **可切换维度**：**所有可展示列都应支持点击表头切换升降序**（这是后台表格通用惯例）
- **场景**：用户管理、内容审核、数据看板
- **额外规则**：可加"我处理的"、"待我处理"作为快捷筛选（不改变排序）

#### 8. 排行榜/TOP N 型

- **默认字段**：`score DESC`（业务定义的核心指标）
- **平局兜底**：优先按次要指标（如注册时间早的更靠前 `created_at ASC`），最后 `id ASC`
- **空值处理**：`NULLS LAST`，但通常上榜条件已过滤空值
- **可切换维度**：可能按维度（日/周/月/总榜）切换；同一维度内排序固定
- **场景**：销量榜、贡献榜、积分榜

#### 9. 树/层级/下拉选项型

- **默认字段**：`sort_order ASC`（人工维护的显示顺序） → `name ASC`（同层级按名称）
- **平局兜底**：`id ASC`
- **空值处理**：`sort_order NULLS LAST`
- **可切换维度**：一般不开放
- **场景**：分类树、部门树、城市选择、下拉字典

#### 10. 高级筛选表单结果集

- **默认字段**：**沿用当前列表业务默认排序**（例如订单列表沿用"下单时间倒序"）
- **可切换维度**：与主列表一致
- **原则**：**筛选只改变结果集，不改变排序方向**。除非筛选中包含"排序"控件（此时以用户选择为准）

### 与筛选表单的组合规则

当排序与筛选并存时，PRD 需明确以下几点：

1. **筛选是否影响排序**：默认不影响。除非筛选中的某项本身与排序含义耦合（如"仅显示未读"这种既是筛选又是排序前置条件）
2. **切换排序时是否保留筛选**：**保留**。用户直觉是"我在当前筛选结果里改变顺序"，而不是重新开始
3. **重置行为**：点击"重置"按钮时，筛选清空 + 排序回归默认
4. **URL / 状态同步**：排序状态是否需要写入 URL query（利于分享和返回时保留）——若产品支持深链，必须写入

### 分页与排序的一致性

分页场景下排序规则必须**稳定且唯一确定顺序**，否则会出现同一条数据在多页重复或漏掉。

**关键做法**：主排序字段必须搭配一个唯一字段（通常是 `id`）作为兜底。

```
❌ 错误：ORDER BY created_at DESC LIMIT 20 OFFSET 20
   （若 created_at 有大量相同值，第 2 页可能重复第 1 页的数据）

✅ 正确：ORDER BY created_at DESC, id DESC LIMIT 20 OFFSET 20
```

对性能敏感或数据量大的列表，PRD 中应提示后端考虑 **游标分页**（keyset pagination），即用 `WHERE (created_at, id) < (last_created_at, last_id)` 代替 `OFFSET`。

### 输出规范

在模块「后端逻辑」章节下，用如下小节结构写出：

```markdown
#### 列表默认排序规则

| 场景 | 主排序字段 | 方向 | 平局兜底 | 空值处理 | 位置 | 可切换维度 |
|---|---|---|---|---|---|---|
| 订单列表 | created_at | DESC | id DESC | 无 | 服务端 | 不开放 |
| 商品搜索结果 | sort_score | DESC | sales_count DESC, id DESC | NULLS LAST | 服务端 | 综合/销量/价格升/价格降/最新 |

**索引建议**：为主排序字段 + 平局兜底字段建立联合索引。
**分页方案**：offset 分页 / 游标分页（建议数据量 > 10w 时用游标）。
**筛选与排序关系**：筛选变化时保留当前排序；排序切换时保留当前筛选；点击"重置"两者一起回到默认。
```

### 排序规则的常见坑（AI 主动避免）

| 坑 | 说明 | 规避 |
|---|---|---|
| 只写"按最新排序" | 未指明字段（是 `created_at` 还是 `updated_at`？），未指明方向 | 显式写出字段名和 `DESC/ASC` |
| 忽略平局兜底 | 主字段有大量相同值时分页会错乱 | 一律加 `id` 作为最后一级 |
| 让前端做分页排序 | 客户端拿不到全量数据 | 分页场景强制服务端排序 |
| 综合排序公式含糊 | "综合"是什么？权重多少？ | 至少写出参与因子清单，权重可留待运营配置 |
| 排序切换清空筛选 | 用户会疑惑"我的条件哪去了" | 明确要求切换时保留筛选 |
| 未考虑软删除 | 已删除数据是否参与排序 | PRD 中明确"仅对 `deleted_at IS NULL` 生效" |
| 时区问题 | `created_at` 存 UTC 还是本地时间 | 存储 UTC，展示按用户时区，排序按存储 |



## 需求逻辑的完整性检查

每个模块讨论完成后，使用以下清单检查完整性：

- [ ] 模块目标是否清晰
- [ ] 核心功能点是否列举完整
- [ ] 交互逻辑是否覆盖正常流程
- [ ] 后端逻辑是否包含状态流转
- [ ] 边界条件是否考虑了异常场景
- [ ] 是否与已确认的其他模块逻辑一致
- [ ] 是否有未明确的模糊点需要追问
- [ ] **模块含列表/筛选表单时，是否已给出完整默认排序规则**（默认字段、方向、平局兜底、空值处理、可切换维度、服务端/客户端）

## 常见问题与处理策略

| 问题 | 处理策略 |
|---|---|
| 用户描述过于笼统 | 提供具体场景选项让用户选择 |
| 功能点之间存在逻辑冲突 | 明确指出冲突，提供2-3种解决方案 |
| 用户无法确定优先级 | 基于用户目标分析依赖关系，给出建议排序 |
| 模块边界模糊 | 按用户目标重新划分，确保每个模块有明确职责 |
| 用户跳过异常场景讨论 | 主动列举2-3个典型异常场景引导思考 |
| 后端逻辑涉及技术实现细节 | 聚焦业务逻辑层面，技术实现留给开发阶段 |
