## 概述

本指南为 Step 1「讨论需求逻辑」提供详细的操作指导。需求逻辑讨论是整个PRD创建流程的基础，目标是明确MVP要做什么、为什么做、怎么做。

---

## Step 1.1 需求范围确定的操作指南

### 信息收集要点

从用户的描述中提取以下关键信息：

| 信息类型 | 说明 | 示例 |
|---|---|---|
| 产品背景 | 为什么要做这个产品/功能 | "现有系统无法满足社区团购需求" |
| 目标用户 | 产品服务于谁 | "社区团长和社区居民" |
| MVP设定逻辑 | 为什么选择这些功能作为MVP | "先验证团购模式可行性" |
| 设计思路 | 整体的产品方向和理念 | "轻量级、社交驱动" |
| 已知约束 | 时间、资源、技术限制 | "3周内上线、仅微信小程序" |

### 需求范围输出规范

输出MVP需求范围时，必须包含以下内容：

1. **纳入MVP的功能清单**：每个功能点附带纳入理由
2. **暂不纳入的功能清单**：每个功能点附带暂缓原因
3. **功能优先级排序**：P0（必须有）/ P1（最好有）/ P2（可以后加）
4. **范围边界说明**：明确什么在范围内、什么不在

### 需求范围判断原则

- MVP聚焦核心价值验证，不追求功能完备
- 优先选择能形成完整用户闭环的功能
- 依赖外部系统或大量开发的功能可考虑暂缓
- 用户明确表示重要的功能必须纳入

## Step 1.2 需求逻辑定义的操作指南

### 模块拆分原则

将MVP需求范围拆分为逻辑模块时，遵循以下原则：

1. **按用户目标拆分**：每个模块对应一个用户目标或任务
2. **高内聚低耦合**：模块内部功能紧密相关，模块间依赖尽量少
3. **粒度适中**：一个模块包含3-8个功能点为宜
4. **命名清晰**：模块名称能直观反映其职责

### 逐模块讨论流程

对每个模块执行以下步骤：

```
1. 告知用户当前模块名称和目标
2. 应用思考框架（见下方）
3. 向用户提出针对性问题
4. 收集用户回答
5. 组织模块需求逻辑内容
6. 检查是否有模糊点，有则追问
7. 展示整理后的模块内容
8. 用户确认 → 进入下一模块
```

### 模块间依赖处理

- 讨论新模块时，回顾已确认模块中与之相关的逻辑
- 如果发现模块间存在依赖关系，明确标注
- 如果发现与已确认模块的逻辑冲突，立即指出并与用户讨论解决

## 模块讨论的思考框架

对每个模块，按以下框架思考并向用户提问：

### 第一层：讲清楚的标准

思考："怎么才算讲清楚了 [模块名] 这件事情？"

判断标准：
- 用户是谁、在什么场景下使用
- 用户要完成什么目标
- 有哪些操作步骤
- 每步操作的输入和输出是什么
- 异常情况如何处理

### 第二层：需要的信息

思考："如果要讲清楚这件事情，我需要告诉用户什么信息？"

按维度收集信息：

| 维度 | 关键问题 |
|---|---|
| 用户与场景 | 谁用？什么时候用？为什么用？ |
| 核心流程 | 主要操作步骤是什么？顺序是什么？ |
| 数据与状态 | 涉及哪些数据？状态如何流转？ |
| 规则与约束 | 有什么业务规则？什么条件下不允许操作？ |
| 异常与边界 | 出错了怎么办？极端情况怎么处理？ |

### 第三层：组织内容

收集到足够信息后，按以下结构组织模块内容：

```markdown
### [模块名]

#### 模块目标
[一句话说明这个模块要解决什么问题]

#### 核心功能点
1. [功能点1]：[说明]
2. [功能点2]：[说明]

#### 交互逻辑
[用户操作流程描述]

#### 后端逻辑
[数据处理、状态流转、业务规则]

#### 边界条件与异常处理
[异常场景及处理方式]
```

## 需求逻辑的完整性检查

每个模块讨论完成后，使用以下清单检查完整性：

- [ ] 模块目标是否清晰
- [ ] 核心功能点是否列举完整
- [ ] 交互逻辑是否覆盖正常流程
- [ ] 后端逻辑是否包含状态流转
- [ ] 边界条件是否考虑了异常场景
- [ ] 是否与已确认的其他模块逻辑一致
- [ ] 是否有未明确的模糊点需要追问

## 常见问题与处理策略

| 问题 | 处理策略 |
|---|---|
| 用户描述过于笼统 | 提供具体场景选项让用户选择 |
| 功能点之间存在逻辑冲突 | 明确指出冲突，提供2-3种解决方案 |
| 用户无法确定优先级 | 基于用户目标分析依赖关系，给出建议排序 |
| 模块边界模糊 | 按用户目标重新划分，确保每个模块有明确职责 |
| 用户跳过异常场景讨论 | 主动列举2-3个典型异常场景引导思考 |
| 后端逻辑涉及技术实现细节 | 聚焦业务逻辑层面，技术实现留给开发阶段 |
