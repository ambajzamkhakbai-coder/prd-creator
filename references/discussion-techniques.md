## 概述

本指南定义PRD创建过程中各讨论环节的提问策略和引导技巧。核心目标：通过高质量提问，帮助用户将模糊的想法转化为清晰的需求描述。

---

## 核心提问框架

### 三层递进提问法

对每个需要讨论的主题，按三层递进提问：

**第一层：场景与目标**
- "用户在什么场景下会用到这个功能？"
- "用户使用这个功能想达成什么目标？"
- "如果不用这个功能，用户现在是怎么做的？"

**第二层：流程与规则**
- "用户完成这个目标的典型步骤是什么？"
- "有什么条件限制吗？什么时候不允许操作？"
- "操作成功和失败分别会怎样？"

**第三层：边界与异常**
- "如果网络断了/数据为空/并发操作会怎样？"
- "有没有特殊的用户角色需要不同处理？"
- "极端数据量（如列表有1000条）时怎么展示？"

## 分步骤提问策略

### Step 1.1 需求范围确定阶段

**开场提问**：

```
请告诉我你要做的产品功能的MVP背景：
1. 为什么要做这个产品/功能？
2. 目标用户是谁？
3. 你设想的MVP核心功能有哪些？
4. 有什么已知的约束条件（时间、资源、技术）？
```

**范围确认提问**：

```
基于你的描述，我梳理了MVP需求范围：
[展示范围清单]

请确认：
1. 纳入MVP的功能是否准确？
2. 暂缓的功能中是否有需要提前的？
3. 有没有遗漏的重要功能？
```

### Step 1.2 模块讨论阶段

**模块开场提问模板**：

```
现在我们来讨论 [模块名] 模块。

完成这个模块内容的撰写，我先思考：
• 怎么才算讲清楚了 [模块名] 这件事情？
• 如果要讲清楚，我需要知道什么信息？

我需要向你了解：
1. [针对该模块的具体问题1]
2. [针对该模块的具体问题2]
3. [针对该模块的具体问题3]
```

**模块确认提问模板**：

```
[展示整理后的模块内容]

请确认这个模块的需求逻辑是否准确：
1. 核心功能点是否有遗漏？
2. 交互逻辑是否符合你的预期？
3. 边界条件的处理是否合理？
```

### Step 2.1 页面结构讨论阶段

**开场提问**：

```
基于已确认的需求逻辑，我梳理了页面结构与导航流程：
[展示页面清单和导航流程图]

请确认：
1. 页面是否完整？有没有遗漏的页面？
2. 导航路径是否符合你的预期？
3. 页面的入口和出口是否正确？
```

### Step 2.2 交互设计讨论阶段

**开场提问**：

```
现在我们来讨论整体的界面和交互设计。

请先告诉我你最关心的界面和交互逻辑是什么？
（抓大放小，先说核心关注点）
```

**补充提问**：

```
基于你关注的核心交互，结合已确认的导航流程，我补充了以下交互设计：
[展示补充内容]

请确认：
1. 核心交互是否符合你的预期？
2. 我补充的交互细节是否合理？
3. 有没有需要调整的地方？
```

## 追问技巧

### 何时追问

以下情况必须追问，不能自行假设：

- 用户使用了模糊词汇（"差不多""大概""可能"）
- 逻辑存在空白（操作步骤之间缺少衔接）
- 存在歧义（可以有多种理解方式）
- 与已确认内容矛盾

### 追问方式

**澄清式追问**：

```
你提到"用户可以管理订单"，具体是指：
A. 查看订单列表
B. 修改订单信息
C. 取消订单
D. 以上全部
```

**场景式追问**：

```
假设一个典型场景：用户下单后想修改收货地址。
在这个场景下，系统应该怎么处理？
```

**反例式追问**：

```
如果用户在支付超时后重新进入订单，应该：
A. 自动取消订单并提示重新下单
B. 保留订单并允许重新支付
C. 其他处理方式
```

## 确认与收敛技巧

### 阶段性总结

每个子步骤完成时，提供简洁的总结：

```
[模块名] 模块讨论完成，核心结论：
1. [结论1]
2. [结论2]
3. [结论3]

是否确认？确认后我们将进入 [下一步骤]。
```

### 差异确认

当用户提出修改时，明确确认修改范围：

```
理解你的修改意见。确认一下修改范围：
- 修改：[具体修改内容]
- 影响：[受影响的其他部分]
- 不变：[确认不变的部分]

这样理解是否正确？
```

### 推进确认

当讨论充分后，主动推进：

```
关于 [当前主题] 的讨论已经比较充分了。
我来整理确认的内容，如果没有新的补充，我们进入下一步。
```

## 特殊情况处理

| 情况 | 处理方式 |
|---|---|
| 用户一次提供大量信息 | 先梳理确认理解是否正确，再分模块讨论 |
| 用户反复修改已确认内容 | 允许修改，但说明影响范围并重新确认受影响部分 |
| 用户说"你决定" | 提供2-3个选项并说明各自优劣，让用户选择 |
| 讨论陷入细节 | 提醒当前步骤的目标，建议先确认框架再深入细节 |
| 用户沉默或只说"好的" | 主动总结当前内容，明确询问是否有补充或调整 |
| 用户需求自相矛盾 | 指出矛盾点，提供解决方案让用户选择 |
