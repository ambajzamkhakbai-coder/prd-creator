# PRD Creator

## 概述

通过结构化的三步方法论，引导用户从模糊的产品想法逐步产出完整的PRD文档。核心原则：**先讨论后产出，每步确认再推进**。整个过程产出三份关键文档：

1. **需求逻辑定义.md** — 明确MVP范围与各模块需求逻辑
2. **界面与交互设计.md** — 确定页面结构、导航流程与交互细节
3. **PRD.md** — 整合前两份文档的完整产品需求文档终稿

## 何时使用

- 用户需要从零开始撰写一份产品需求文档
- 用户有产品想法但尚未结构化为正式PRD
- 用户希望系统化地梳理需求逻辑和交互设计
- 用户提到"写PRD""需求文档""产品需求"等关键词

## 快速开始

```
我想做一个社区团购小程序的PRD
```

```
帮我写一份在线教育平台MVP的产品需求文档，核心功能是直播课和作业系统
```

## 工作流程

### 总体流程

```
Step 1: 讨论需求逻辑 → 输出 需求逻辑定义.md
    ↓（用户确认）
Step 2: 确定界面+交互设计 → 输出 界面与交互设计.md
    ↓（用户确认）
Step 3: 整合输出PRD终稿 → 输出 PRD.md
```

**关键约束**：每个步骤和子步骤完成后必须暂停，等待用户明确确认后才进入下一步。用户说"确认""没问题""继续"等表示通过；用户提出修改意见则先修改再确认。

---

### Step 1: 讨论需求逻辑

目标：通过讨论明确MVP需求范围和各模块需求逻辑，最终输出「需求逻辑定义.md」。

#### Step 1.1: 需求范围确定

1. 询问用户提供产品功能的MVP背景、设定逻辑、设计思路
2. 基于用户提供的信息，输出MVP需求范围，包含：
   - 需求范围清单（包含的功能点）
   - 暂不纳入MVP的功能点及原因
   - 每个纳入功能点的纳入理由
3. 与用户讨论需求范围，根据反馈调整
4. 用户确认需求范围后，进入 Step 1.2

**输出格式**：

```
## MVP 需求范围

### 纳入MVP的功能
| 功能点 | 纳入理由 |
|---|---|
| [功能] | [为什么MVP需要这个] |

### 暂不纳入MVP的功能
| 功能点 | 暂缓原因 |
|---|---|
| [功能] | [为什么MVP阶段不需要] |
```

#### Step 1.2: 需求逻辑定义（逐模块讨论）

在确认的MVP需求范围基础上，逐模块讨论需求逻辑。**一次只讨论一个模块**。

对每个模块，执行以下流程：

1. 告知用户当前讨论的模块名称
2. 按照以下思考框架引导讨论：
   - "完成 [模块名] 模块内容的撰写。我先思考："
   - "怎么才算讲清楚了这件事情？"
   - "如果要讲清楚这件事情，我需要告诉你什么信息？"
   - 然后向用户提出针对性问题
3. 收到用户回复后，组织该模块的需求逻辑内容
4. 如果仍有模糊之处，继续追问直到清晰
5. 将该模块的讨论结果整理后展示给用户确认
6. 用户确认后，进入下一个模块

**每个模块的讨论结果应包含**：
- 模块目标
- 核心功能点及逻辑说明
- 交互逻辑（用户操作流程）
- 后端逻辑（数据处理、状态流转）
- 边界条件与异常处理

#### Step 1.3: 生成需求逻辑定义.md

所有模块讨论完毕并确认后：
1. 按照需求逻辑定义模板（参见 [requirement-logic-template.md](assets/requirement-logic-template.md)）组织完整内容
2. 将MVP需求范围框架 + 各模块的需求逻辑定义整合
3. 使用 Write 工具生成「需求逻辑定义.md」文件
4. 展示给用户最终确认

**暂停点**：用户确认「需求逻辑定义.md」后，才进入 Step 2。

---

### Step 2: 确定界面+交互设计

目标：在需求逻辑明确的基础上，讨论确定整体界面与交互设计，最终输出「界面与交互设计.md」。

#### Step 2.1: 讨论页面结构与导航流程

1. 基于已确认的需求逻辑定义，梳理产品涉及的页面清单
2. 绘制页面之间的导航流程（使用 Mermaid 语法）
3. 标注关键页面的人口和出口
4. 按照页面结构与导航流程模板（参见 [page-navigation-template.md](assets/page-navigation-template.md)）输出初稿
5. 展示给用户，如果与用户预想差距较大，通过讨论纠正
6. 用户确认后，进入 Step 2.2

**输出格式**：包含页面清单表 + Mermaid 导航流程图 + 页面说明

#### Step 2.2: 讨论整体界面+交互设计

1. 询问用户最关心的界面和交互逻辑（抓大放小，用户先说核心关注点）
2. 结合已确认的页面结构与导航流程，完善整体的界面和交互设计描述
3. 针对每个关键页面，描述：
   - 页面布局结构
   - 核心交互元素及操作反馈
   - 状态变化与数据流转
   - 异常场景的交互处理
4. 给出反馈意见和优化建议
5. 与用户讨论直到达成一致

#### Step 2.3: 生成界面与交互设计.md

1. 按照界面与交互设计模板（参见 [ui-interaction-template.md](assets/ui-interaction-template.md)）组织完整内容
2. 将页面结构与导航流程 + 整体界面与交互设计整合
3. 使用 Write 工具生成「界面与交互设计.md」文件
4. 展示给用户最终确认

**暂停点**：用户确认「界面与交互设计.md」后，才进入 Step 3。

---

### Step 3: 整合输出PRD终稿

目标：将「需求逻辑定义.md」和「界面与交互设计.md」整合为完整的PRD文档。

1. 读取已确认的「需求逻辑定义.md」和「界面与交互设计.md」
2. 按照PRD模板（参见 [prd-template.md](assets/prd-template.md)）组织完整内容
3. 整合逻辑：
   - 从「需求逻辑定义.md」提取：产品背景、MVP范围、功能需求、后端逻辑
   - 从「界面与交互设计.md」提取：页面结构、导航流程、交互设计
   - 补充：非功能需求、数据需求、风险与依赖等章节
4. 使用 Write 工具生成「PRD.md」文件
5. 展示给用户最终确认

## 约束与注意事项

- **严格分步确认**：每个子步骤完成后必须暂停等待用户确认，不得跳过或自动推进
- **一次一个模块**：Step 1.2 中每次只讨论一个模块，确认后再进入下一个
- **不替用户做决定**：遇到需要产品决策的问题时，向用户提问而非自行假设
- **保持文档一致性**：后续步骤的内容必须与已确认的前序文档保持一致，如有冲突需与用户确认
- **Mermaid图表**：所有流程图、导航图使用 Mermaid 语法，确保在 Markdown 中可渲染
- **文件输出路径**：所有生成的 .md 文件保存到用户当前工作目录
- **讨论引导策略**：参考 [discussion-techniques.md](references/discussion-techniques.md) 中的提问技巧

## 错误处理

- **用户需求过于模糊**：提供2-3个具体场景选项让用户选择，而非反复追问
- **模块间逻辑冲突**：发现冲突时明确指出，提供解决方案让用户选择
- **用户跳步请求**：提醒用户当前步骤尚未完成，说明完成当前步骤的必要性
- **用户修改已确认内容**：允许修改，但需说明对后续步骤的影响并重新确认受影响的部分
- **需求范围过大**：建议拆分优先级，先完成核心MVP再扩展
